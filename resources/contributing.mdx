We welcome contributions to Reze AI agent! This guide will help you get started with contributing code, documentation, bug reports, and feature requests.

## Getting started

<Steps>
  <Step title="Fork the repository">
    Create your own fork of the repository:
    ```bash
    # Visit https://github.com/Nneji123/reze-agent
    # Click "Fork" button
    ```
  </Step>
  
  <Step title="Clone your fork">
    ```bash
    git clone https://github.com/YOUR_USERNAME/reze-agent.git
    cd reze-agent
    ```
  </Step>
  
  <Step title="Set up development environment">
    Install dependencies with development tools:
    ```bash
    # Install uv package manager
    pip install uv
    
    # Install all dependencies including dev tools
    uv sync --dev
    ```
  </Step>
  
  <Step title="Create feature branch">
    ```bash
    git checkout -b feature/AmazingFeature
    ```
  </Step>
</Steps>

<Note>
  Always create a new branch for your changes. Never commit directly to `main`.
</Note>

## Development workflow

### Project structure

```

├── src/
│   ├── api/              # FastAPI routes and schemas
│   │   ├── routers/      # API endpoints
│   │   └── schemas/      # Pydantic models
│   ├── services/         # Business logic
│   │   ├── ai.py         # PydanticAI agent
│   │   ├── rag.py        # RAG integration
│   │   ├── memvid.py     # Knowledge base
│   │   ├── resend.py     # Email service
```

<Tabs>

### Code style and linting

<AccordionGroup>
  <Accordion title="Ruff">
    We use **Ruff** for linting and formatting (>= 0.0.263):
    
    ```bash
    # Check for issues
    uv run ruff check .
    
    # Fix auto-fixable issues
    uv run ruff check --fix .
    
    # Format code
    uv run ruff format .
    ```
    
    Configuration in `pyproject.toml:28-46`
  </Accordion>
  
  <Accordion title="Type checking">
    We use **Pyright** for type checking:
    
    ```bash
    # Install pyright
    npm install -g pyright
    
    # Run type checker
    pyright src/
    ```
    
    Configuration in `pyproject.toml:28-46`
  </Accordion>
</AccordionGroup>

### Testing

<Steps>
  <Step title="Write tests">
    Add tests in `tests/` directory:
    ```python
    # tests/test_resend.py
    import pytest
    from src.services.resend import ResendService
    
    @pytest.mark.asyncio
    async def test_send_email():
        service = ResendService()
        # Test implementation
    ```
  </Step>
  
  <Step title="Run tests">
    ```bash
    # Run all tests
    uv run pytest
    
    # Run with coverage
    uv run pytest --cov=src --cov-report=html
    
    # Run specific test file
    uv run pytest tests/test_resend.py
    ```
  </Step>
  
  <Step title="View coverage">
    Open `htmlcov/index.html` in your browser to see coverage report
  </Step>
</Steps>

<Info>
  Test configuration is in `pyproject.toml:48-66`. We use pytest with asyncio support.
</Info>

### Test markers

Organize tests with markers:

```python
@pytest.mark.unit
def test_email_validation():
    # Unit test
    pass

@pytest.mark.integration
async def test_resend_api():
    # Integration test
    pass

@pytest.mark.slow
async def test_full_workflow():
    # Slow end-to-end test
    pass
```

Run specific markers:
```bash
pytest -m unit        # Only unit tests
pytest -m "not slow"  # Skip slow tests
```

## Making changes

### Adding new features

<Steps>
  <Step title="Plan your feature">
    - Check existing issues for duplicates
    - Open a discussion or issue to propose the feature
    - Get feedback before implementing
  </Step>
  
  <Step title="Implement the feature">
    - Follow existing code patterns
    - Add type hints to all functions
    - Use async/await for I/O operations
    - Add logging with loguru
  </Step>
  
  <Step title="Write tests">
    - Add unit tests for new functions
    - Add integration tests for API changes
    - Ensure 80%+ code coverage
  </Step>
  
  <Step title="Update documentation">
    - Update README.md if needed
    - Add docstrings to new functions
    - Update API documentation
  </Step>
</Steps>

### Fixing bugs

<Steps>
  <Step title="Reproduce the bug">
    - Create a test that fails
    - Document the expected vs actual behavior
  </Step>
  
  <Step title="Fix the issue">
    - Make minimal changes to fix the bug
    - Ensure the test now passes
    - Check for similar issues elsewhere
  </Step>
  
  <Step title="Add regression test">
    - Ensure the bug can't happen again
    - Test edge cases
  </Step>
</Steps>

### Code guidelines

<AccordionGroup>
  <Accordion title="Type hints">
    Always use type hints:
    ```python
    # Good
    async def send_email(
        to: str,
        subject: str,
        html_content: str
    ) -> EmailResponse:
        ...
    
    # Bad
    async def send_email(to, subject, html_content):
        ...
    ```
  </Accordion>
  
  <Accordion title="Error handling">
    Use structured error handling:
    ```python
    try:
        result = await operation()
        logger.success(f"Operation succeeded: {result}")
        return result
    except SpecificError as e:
        logger.error(f"Operation failed: {e}")
        raise
    ```
  </Accordion>
  
  <Accordion title="Logging">
    Use loguru for logging:
    ```python
    from loguru import logger
    
    logger.info("Starting operation")
    logger.success("Operation completed")
    logger.warning("Potential issue detected")
    logger.error("Operation failed")
    ```
  </Accordion>
  
  <Accordion title="Docstrings">
    Use Google-style docstrings:
    ```python
    async def search_documents(
        query: str,
        k: int = 5,
        mode: str = "hybrid"
    ) -> list[dict]:
        """Search documents in Memvid.
        
        Args:
            query: Search query string
            k: Number of results to return
            mode: Search mode (lex, sem, or hybrid)
        
        Returns:
            List of search result dictionaries
        """
        ...
    ```
  </Accordion>
</AccordionGroup>

## Submitting changes

### Commit messages

Write clear, descriptive commit messages:

<Tabs>
  <Tab title="Good examples">
    ```bash
    git commit -m "Add email attachment support to ResendService"
    git commit -m "Fix Memvid corruption recovery logic"
    git commit -m "Update FastAPI to v0.110.0 for security patch"
    ```
  </Tab>
  
  <Tab title="Bad examples">
    ```bash
    git commit -m "fix bug"              # Too vague
    git commit -m "WIP"                  # Not descriptive
    git commit -m "asdfasdf"             # Meaningless
    ```
  </Tab>
</Tabs>

### Creating a pull request

<Steps>
  <Step title="Commit your changes">
    ```bash
    git add .
    git commit -m "Add some AmazingFeature"
    ```
  </Step>
  
  <Step title="Push to your fork">
    ```bash
    git push origin feature/AmazingFeature
    ```
  </Step>
  
  <Step title="Open pull request">
    - Go to your fork on GitHub
    - Click "New Pull Request"
    - Select your feature branch
    - Fill out the PR template
  </Step>
  
  <Step title="Address review feedback">
    - Respond to comments
    - Make requested changes
    - Push updates to the same branch
  </Step>
</Steps>

### Pull request checklist

Before submitting, ensure:

<Accordion title="Code quality">
  - [ ] Code follows project style guidelines
  - [ ] Ruff linter passes: `uv run ruff check .`
  - [ ] Type checking passes: `pyright src/`
  - [ ] All tests pass: `uv run pytest`
  - [ ] Added tests for new features
  - [ ] Code coverage maintained or improved
</Accordion>

<Accordion title="Documentation">
  - [ ] Added docstrings to new functions
  - [ ] Updated README if needed
  - [ ] Updated API documentation
  - [ ] Added code comments for complex logic
</Accordion>

<Accordion title="Git">
  - [ ] Meaningful commit messages
  - [ ] Branch is up to date with main
  - [ ] No merge conflicts
  - [ ] Commits are logical and atomic
</Accordion>
